開場：你有沒有想過⋯⋯

大家有沒有遇過這種事？

公司裡有個超難用的表單，每次填都想翻桌。或是某個流程明明可以自動化，但就是沒有工程師有空幫你做。你心裡想著「如果我自己會寫程式就好了」，然後就⋯⋯算了。

今天這堂課要告訴你一件事：你不需要學會寫程式，你只需要學會跟 AI 講話。

但在開始之前，我們先聊一個重要的觀念。

在 AI 時代，什麼最珍貴？

大家工作上一定遇過這種事——突然靈光一閃有個好點子，但手上事情太多根本沒空做。好不容易有空了，打開電腦，Line 響、Email 來、被拉進會議，專注力瞬間歸零。

現在好消息是，我們有 AI 了。你可以叫 AI 幫你動手做事。但 AI 也不是免費的許願池——每一次你跟 AI 對話，它都在消耗一種叫 Token 的東西。

Token 是什麼？你可以把它想成 AI 的字數額度。你打字給它要花 Token，它回覆你也要花 Token。講越多、來回越多次，Token 就燒越快。免費版一天能用的量有限，付費版也是按量計費的。

所以你看，在這個 AI 時代，真正珍貴的東西長這樣：

創意很貴——但你有時間去做嗎？
時間很貴——但你能專注不被打斷嗎？
專注力很貴——但你跟 AI 講的每句話都在燒錢，你確定沒在浪費？

今天這堂課就一個目的：讓你學會怎麼把這些珍貴的東西，花在刀口上。

什麼是 Vibe Coding？

傳統的開發流程是這樣的：

你有個想法 → 寫成需求單 → 等工程師排程 → 開發 → 測試 → 修 Bug → 再測試 → 上線

這個流程快的話幾週，慢的話幾個月。

Vibe Coding 的流程是這樣的：

你有個想法 → 用自然語言跟 AI 描述 → AI 幫你寫程式 → 你看成果 → 不滿意就再講一次 → 調整到滿意 → 上線

幾個小時就能完成。

Vibe Coding 不是學寫程式，是學會「用你的話告訴 AI 你要什麼」。

心態轉換：從球員變教練

學 Vibe Coding，最大的改變不是技術，是心態。

以前你可能覺得：「我不會寫程式，所以我做不到。」
現在你要換成：「我不需要自己寫，我只需要指揮 AI 幫我做。」

你的新角色有三個重點：

1. 你不需要懂程式語法，但你要懂得「把需求說清楚」。
AI 的能力很強，但它不會讀心術。你講得越清楚，它做得越好。

2. 你的價值在於「知道要做什麼」，不是「知道怎麼寫」。
你比 AI 更了解你的工作、你的痛點、你的使用者。這才是真正值錢的東西。

3. 錯了不用怕，隨時可以重來。
AI 不會累、不會生氣、不會覺得你煩。做錯了就再跟它說一次，但你要學會判斷「哪裡錯了」。

現在的 AI 開發能做到什麼？

你可能覺得 AI 做出來的東西很陽春。來看幾個真實的例子：

  • 用 AI 做一個請假天數計算器，輸入到職日自動算特休
  • 用 AI 生成一個產品比較表網頁，給客戶看的那種
  • 用 AI 寫一個自動整理 Excel 的小工具，每月報表不用再手動搬資料

這些不是工程師花幾天做出來的，是普通人跟 AI 對話幾個小時的成果。

今天這堂課結束的時候，你也會做出一個可以上線的工具。

Google 提供了一整套 AI 開發工具，從最簡單到最專業，剛好對應你的學習階段：

Ch3 寫出好 Prompt
▼

Part A — 概念篇

為什麼 Prompt 很重要？

還記得 Ch1 講的 Token 嗎？每次你跟 AI 對話都在消耗資源和時間。如果你的 Prompt 寫得模糊，AI 回的東西不是你要的，你就得再來一次——時間又浪費了一輪。

模糊的 Prompt

「幫我做一個表格。」

AI 反問：「什麼表格？幾欄？內容？」
→ 又要花一次對話補充

精準的 Prompt

「幫我做一個員工特休計算表，欄位有：姓名、到職日、年資、特休天數、已用天數、剩餘天數。先給我 5 筆範例資料。」

→ AI 直接生出你要的東西，一次到位

Prompt 的品質 = 你省下的時間。

Part B — 練習篇

範例一：請 AI 寫一封公告信

Before（模糊版）

「幫我寫一封公告。」

After（精準版）

「你是公司的行政助理。我們公司將在 1/17（五）晚上 6 點在晶華酒店舉辦尾牙。請幫我寫一封全公司的公告 Email。內容要包含：時間地點、交通方式、dress code（smart casual）、報名截止日（1/10）。語氣輕鬆活潑但不失專業，長度控制在 200 字以內。」

範例二：請 AI 做一個計算工具

Before（模糊版）

「幫我做一個計算機。」

After（精準版）

「我想做一個員工特休天數計算器的網頁工具。功能：使用者輸入到職日，系統自動計算年資和對應的特休天數（依照台灣勞基法規定）。介面：一個日期輸入框、一個「計算」按鈕、下方顯示結果。風格：簡潔乾淨，白色背景，藍色按鈕。」

範例三：請 AI 整理會議紀錄

Before（模糊版）

「幫我整理會議紀錄。」

After（精準版）

「以下是今天行銷部週會的逐字稿（貼上內容）。請幫我整理成會議紀錄，格式：會議名稱、日期時間、出席者、討論事項（條列＋結論）、待辦事項（負責人＋截止日）。語氣正式但簡潔。」

Ch4 SCOPE 判斷法
▼

為什麼要判斷可行性？

學到這裡，你已經會跟 AI 對話了，也知道怎麼寫好的 Prompt。接下來你一定會開始有很多想法：「我想做這個！我想做那個！」

但不是每個想法都適合馬上開始做。如果你不先判斷就直接做，結果通常是：花了兩小時，做出一個半成品，然後卡住。

所以在動手之前，先用 SCOPE 判斷法 花三分鐘做個快速檢查。

實例：用 SCOPE 檢查「IG 留言抽獎器」

題目：做一個 IG 留言抽獎器，可以貼上留言內容，自動解析帳號、去重複，設定人數後隨機抽獎。

檢查項目分析結果
S — Size「一個可以貼上 IG 留言、自動抽出中獎者的網頁工具」→ 一句話搞定✅
C — Clear貼什麼進去？怎麼辨識帳號？抽幾個人？→ 都能講清楚✅
O — Output最終是一個 HTML 網頁，顯示中獎者名單✅
P — Parts①輸入框 ②解析帳號 ③去重複 ④設人數 ⑤抽獎動畫 → 5 個⚠️
E — Example可以參考各種線上抽獎工具✅
結論：這個題目可以做！Parts 有 5 個在邊界上，我們用半成品的方式降低複雜度，讓大家專注在最有趣的部分。
Ch5 READ 除錯法
▼

壞掉是正常的

在你跟 AI 一起做東西的過程中，一定會遇到東西壞掉的時候。按鈕按了沒反應、畫面跑版、出現紅色錯誤訊息⋯⋯

這不是你的問題，也不是 AI 的問題。程式開發本來就是這樣——做出來、壞掉、修好、再壞掉、再修好。專業的工程師每天也在做一樣的事。

示範一：抽獎按鈕按了沒反應

R — Read

按 F12 → Console → 看到紅字：
winnerCount is not defined
不用看懂，先複製起來

E — Explain

貼回 AI Studio：「我按下抽獎按鈕沒反應，Console 出現這個錯誤⋯⋯我期待會抽出中獎者」

A — Ask

AI 說：「變數名稱對不上」
你追問：「為什麼名字不同就會壞？」
AI 解釋原因

D — Do again

改一致 → 預覽更新 → 再按一次 → 成功抽出中獎者了！

示範二：解析出來的帳號數量是 0

R — Read

沒有紅色錯誤，但畫面顯示「共 0 位參加者」。線索在畫面上，不在 Console

E — Explain

貼回 AI Studio：「我貼上了留言但顯示 0 人。我貼的內容長這樣（附範例）」

A — Ask

AI：「解析邏輯用 @ 辨識帳號，但你的格式裡沒有 @」
你追問：「能加預覽功能嗎？」

D — Do again

更新邏輯 → 重新貼上 → 顯示 15 位參加者 → 抽獎成功！

準備：拿你的免費 Gemini API Key

在開始寫程式之前，你需要先拿一把「通行證」——API Key。這是你的 Vue 專案用來跟 Google AI 說話的鑰匙，完全免費。

如何取得 Gemini API Key（5 分鐘內搞定）

1. 打開瀏覽器，前往：

https://aistudio.google.com/apikey

2. 用你的 Google 帳號登入（和 Gmail 同一個帳號就好）。

3. 點擊藍色的 「Create API key」 按鈕。

4. 選擇 「Create API key in new project」（如果你沒有現有專案），或者選你已有的專案。

5. 畫面會出現一串像這樣的文字：AIzaSy...（很長的一串英文加數字）。這就是你的 API Key。

6. 按 「Copy」 把它複製起來，貼到記事本暫時存好。等一下設定 Vue 專案時會用到。

重要：API Key 就像密碼，不要貼到公開的地方（不要放進 GitHub 公開 repo 的程式碼裡）。等一下會教你怎麼安全地使用它。
免費額度夠用嗎？ 夠！Google 的免費方案每分鐘可以呼叫 15 次，今天課程完全用不完。

開始實作：保母級步驟教學

接下來的每一步都會詳細告訴你要按哪裡、要打什麼字。跟著做就對了。

Step 1：打開 Google AI Studio 並進入 Build Mode

1-1. 打開瀏覽器（Chrome 最佳），在網址列輸入：

https://aistudio.google.com

1-2. 如果還沒登入 Google 帳號，畫面會請你登入。用你平常用的 Gmail 帳號登入就好。

1-3. 第一次進入會看到「Terms of Service」（服務條款），勾選同意，按 Continue。

1-4. 登入後你會看到首頁。看向左邊側邊欄，你會看到幾個選項：Home、Playground、Build、Dashboard、Documentation。

1-5. 點擊 「Build」。

1-6. 你現在進入了 Build Mode。畫面會分成左右兩邊：

  • 左邊：Prompt 輸入區（你跟 AI 講話的地方）
  • 右邊：即時預覽區（AI 做出來的東西會顯示在這裡）
小技巧：如果你不想打字，左下角有一個麥克風按鈕，按住說話就能用語音輸入 Prompt！

Step 2：把半成品程式碼貼進去

2-1. 打開講師提供的半成品程式碼檔案（附件中的 HTML 檔案），用文字編輯器打開（記事本就可以）。

2-2. 全選所有內容（Ctrl+A），複製（Ctrl+C）。

2-3. 回到 AI Studio 的 Build Mode，在左邊的 Prompt 輸入框中，貼上以下文字：

以下是我目前的網頁程式碼，請幫我把它跑起來，然後等我告訴你要修改什麼。

（在這裡按 Ctrl+V 貼上你剛才複製的程式碼）

2-4. 按下輸入框右邊的送出按鈕（或按 Ctrl+Enter）。

2-5. 等幾秒鐘，AI 會開始處理。處理完畢後，右邊的預覽區會直接顯示你的半成品網頁。你應該可以看到基本的介面出現了。

如果右邊什麼都沒出現：不要慌。在左邊的輸入框追加一句：「請幫我用 HTML/CSS/JavaScript 在右邊的預覽區跑起來。」然後再送出。

Step 3：跟 AI 說你想改什麼

現在半成品已經在右邊跑起來了。接下來就是 Vibe Coding 的核心——用自然語言跟 AI 說你想改什麼。

記住 Ch3 學的 Prompt 結構：角色 + 情境 + 任務 + 格式 + 限制

以下是各選項的範例 Prompt，你可以直接複製使用，也可以自己修改：

如果你選的是「名單隨機分組器」：

範例 Prompt 1 — 加上分組動畫：
我想幫這個名單分組器加上動畫效果。按下「開始分組」按鈕之後，不要立刻顯示結果，而是先跑一個 3 秒的洗牌動畫（名字快速閃動），然後再慢慢顯示每一組的結果。動畫風格要活潑有趣。
範例 Prompt 2 — 改外觀顏色：
請把整個頁面的風格改成深色主題：背景色用深灰色 (#1a1a2e)，文字用白色，按鈕用亮紫色 (#6C63FF)，每一組結果用不同的淡色卡片呈現。整體要有現代科技感。
範例 Prompt 3 — 加新功能：
請在結果區下面加一個「重新分組」按鈕，按下去會重新隨機分配（但名單不變）。另外再加一個「匯出結果」按鈕，按下去可以把分組結果複製到剪貼簿。

如果你選的是「IG 留言抽獎器」：

範例 Prompt 1 — 加抽獎動畫：
我想在按下「開始抽獎」按鈕之後，有一個轉盤動畫效果。轉盤上面顯示所有參加者的帳號名稱，轉盤轉動 3 秒後慢慢停下來，停到中獎者的名字上。然後跳出撒紙花的慶祝動畫，同時顯示中獎者名單。
範例 Prompt 2 — 加排除功能：
請在頁面上加一個「排除帳號」的輸入框，讓我可以輸入不參加抽獎的帳號（例如我自己的帳號或小編帳號），用逗號分隔。這些帳號在抽獎時要自動被排除。
範例 Prompt 3 — 改成活動風設計：
請把整個頁面改成派對活動風格：背景用漸層色（從粉紅到橘色），標題用大大的手寫風文字，按鈕用圓角加上陰影，整體要讓人感覺是一個歡樂的抽獎活動頁面。頁面上方加一個「活動名稱」的輸入框讓使用者自訂。

Step 4：看預覽、測試、繼續調整

4-1. 每次你送出 Prompt 之後，右邊的預覽區會自動更新。仔細看看結果對不對。

4-2. 試試看功能能不能用：按按鈕、輸入資料、測試各種情境。

4-3. 如果有地方不滿意，在左邊的對話框繼續跟 AI 說。例如：

「動畫速度太快了，可以改成 5 秒嗎？」
「按鈕的顏色不好看，換成綠色。」
「結果的字太小了，放大到 18px。」
「加一個頁尾，顯示 © 2026 我的名字。」

4-4. 一次只改一件事。改完確認 OK 再改下一個。不要一口氣講五個需求。

4-5. 如果畫面壞掉了（按鈕沒反應、畫面空白、跑版），用 Ch5 的 READ 除錯法：

  1. 按 F12 打開 Console，看有沒有紅色錯誤訊息
  2. 把錯誤訊息複製起來
  3. 貼回左邊的對話框，跟 AI 說明情況
  4. AI 修好之後，重新測試
超實用句型：如果你改到一半覺得亂掉了，可以跟 AI 說：
「目前的程式碼太亂了，請幫我整理一下，保留目前所有功能，但把程式碼重新排列乾淨。」

4-6. 功能都確認 OK 之後，最後再跟 AI 說一件事：

最後一個 Prompt — 請 AI 幫你建立自動部署設定：
我要把這個 Vue 專案部署到 GitHub Pages，請幫我建立 .github/workflows/deploy.yml 的 GitHub Actions 自動部署設定檔。我的 GitHub repo 名稱是「你的repo名稱」。

AI 會幫你生成完整的 deploy.yml，你只需要把它複製到專案裡對應的位置就好，不用自己手動寫。

Step 5：滿意之後，把程式碼存下來

5-1. 改到你滿意了之後，點擊右邊預覽區上方的 「Code」 分頁，可以看到 AI 幫你寫的完整程式碼。

5-2. 把程式碼複製下來，更新你的 Vue 專案對應的檔案。

接下來要做什麼？ 程式碼確認沒問題之後，我們會用 GitHub Actions 把它自動部署到網路上。只要 push 一次，GitHub 就會幫你跑完所有部署流程——完全不用手動操作。設定只需要做一次，之後改完程式碼 push 就自動更新。

部署到 GitHub Pages：保母級步驟教學

你的 Vue 專案只需要設定一次，之後每次 push 程式碼，GitHub 就會自動幫你重新部署。全程免費。

開始之前，確認你已經有：
  • GitHub 帳號（還沒有的看 Step 0）
  • Vue 專案已在電腦上建立完成
  • Gemini API Key（已在上一節取得，若專案有使用 AI 功能才需要）

Step 0：註冊 GitHub 帳號（已有帳號跳過這步）

0-1. 打開瀏覽器，前往：

https://github.com

0-2. 點擊右上角的 「Sign up」。

0-3. 輸入 Email、設定密碼、選擇使用者名稱。使用者名稱會出現在你的網址裡（例如 你的名字.github.io），取一個好記的英文名稱。

0-4. 完成驗證流程，到 Email 收驗證信，點擊確認連結。帳號就建立好了。

Step 1：修改 vite.config.ts — 設定部署路徑

GitHub Pages 的網址結構是 https://帳號.github.io/repo名稱/，所以你必須告訴 Vite 在打包時要加上這個路徑前綴，否則頁面會找不到資源、出現空白或 404。

1-1. 打開專案根目錄的 vite.config.ts 檔案。

1-2. 在 defineConfig({...}) 裡加入 base 這一行：

import { defineConfig } from 'vite'
// 其他 import 保留不動...

export default defineConfig({
  base: '/你的repo名稱/',  // ← 加這行
  // 其他設定保留不動...
})
注意：base 的值必須和你的 GitHub Repository 名稱完全一致，包含大小寫。例如 repo 叫 tomato-clock，這裡就寫 '/tomato-clock/'。

1-3. 存檔。

Step 2：建立自動部署設定檔

這個設定檔會告訴 GitHub：每次你 push 新程式碼，就自動幫你 build 並部署到網路上。只需要建立一次，之後完全自動。

2-1. 在你的專案根目錄，依序建立這些資料夾和檔案：

專案根目錄 → 新增資料夾 .github → 在裡面新增資料夾 workflows → 在裡面新增檔案 deploy.yml

2-2. 把以下內容完整貼進 deploy.yml：

name: Deploy static content to Pages

on:
  push:
    branches: ['main']

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: 'pages'
  cancel-in-progress: true

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Set up Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm install
      - name: Build
        run: npm run build
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}  # 沒用到 AI 功能可刪這行
      - name: Setup Pages
        uses: actions/configure-pages@v4
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: './dist'
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

2-3. 存檔。

這份設定在做什麼？ 每次你 push 到 main 分支，GitHub 會自動：安裝套件 → 執行 npm run build 打包 → 把 dist 資料夾的內容部署到網路上。你完全不用手動操作。

Step 3：把 API Key 安全地存進 GitHub （專案有用 AI 功能才需要）

API Key 不能直接寫在程式碼裡再 push 上去，那樣全世界都看得到。GitHub Secrets 就是專門存這種機密資料的地方——只有你的部署流程能讀取，其他人看不到。

3-1. 進入你的 GitHub Repository 頁面，點擊上方的 「Settings」。

3-2. 在左邊側邊欄找到 「Secrets and variables」，點開後選 「Actions」。

3-3. 點擊 「New repository secret」。

3-4. 填寫：

  • Name：GEMINI_API_KEY（必須完全一致，大小寫都算）
  • Value：貼上你的 API Key（就是那串 AIzaSy...）

3-5. 點擊 「Add secret」。存好了，之後部署時 GitHub 會自動讀取這個值。

Step 4：在 GitHub 開啟 Pages 功能

4-1. 進入你的 GitHub Repository 頁面，點擊上方的 「Settings」。

4-2. 在左邊側邊欄找到 「Pages」，點擊它。

4-3. 在 「Build and deployment」 區塊，找到 「Source」 下拉選單。

4-4. 把 Source 改選為 「GitHub Actions」。

注意：這裡要選 「GitHub Actions」，不是選 branch。因為我們用的是自動部署腳本，不是直接從 branch 部署。選錯會導致網站沒辦法正常上線。

4-5. 設定會自動儲存。

Step 5：Push！觸發第一次自動部署

所有設定都好了。現在只要把程式碼推上 GitHub，後面的事 GitHub 會自動幫你做——你不需要手動跑任何指令。

5-1. 在你的 AI 開發工具（Cursor 或 VS Code）的對話框輸入：

請幫我把目前所有的更動 commit 並 push 到 GitHub。commit message 寫「deploy to github pages」。

AI 會自動幫你完成 commit 和 push，你只要確認一下有沒有跑成功就好。

5-2. Push 完之後，打開 GitHub Repository 頁面，點擊上方的 「Actions」 頁籤。

5-3. 你會看到一個正在執行的工作，旁邊有黃色旋轉圈圈。等它變成綠色勾勾，就代表部署成功了（通常 2–3 分鐘）。

▲ 每一行都有綠色勾勾代表部署成功。左側「Deploy static content to Pages」是你的自動部署工作流程。

5-4. 部署完成後，你的網站網址是：

https://你的帳號.github.io/你的repo名稱/

把這個網址分享給任何人，他們用手機或電腦打開就能直接使用。之後每次你改完程式碼，一樣叫 AI 幫你 commit 和 push，網站就會自動在幾分鐘內更新。

檢查網站時，記得清快取！ 瀏覽器會把舊版本的網頁記在快取裡，導致你看到的還是舊畫面，以為部署沒成功。正確的做法：
  • 按 Ctrl + Shift + R（Mac 用 Cmd + Shift + R）強制重新整理並清快取
  • 或是直接開一個無痕視窗貼上網址，完全沒有快取問題
常見問題：
  • Actions 顯示紅色叉叉 → 點進去看錯誤訊息，最常見的原因是 vite.config.ts 的 base 名稱打錯，或是 deploy.yml 縮排跑掉
  • 網站顯示空白或 404 → 確認 base 的值和 repo 名稱完全一致
  • 改了程式碼但網站沒更新 → 等 2-3 分鐘再重新整理，或去 Actions 頁籤確認部署有跑完
  • API 呼叫失敗 → 確認 GitHub Secrets 的名稱是 GEMINI_API_KEY（全大寫），且 Value 貼的是完整的 key

恭喜你完成了！

下次工作上遇到什麼想自動化的事情，別再等工程師了——打開 Google AI Studio，SCOPE 檢查一下，然後動手做。
